## 2024-05-01 - Prevent Mass Assignment and Implement Authorization on Support Tickets
**Vulnerability:** The `/api/support/tickets` and `/api/support/tickets/:id` endpoints allowed mass assignment, where users could manually inject `userId` and `status` fields via the `req.body`. Additionally, the `PATCH /api/support/tickets/:id` route lacked an ownership/authorization check, enabling any authenticated user to update any support ticket, including modifying its `userId` and `status`.
**Learning:** These endpoints failed to validate or strip sensitive keys from incoming user requests, relying entirely on trust rather than using a schema for validation. Furthermore, fetching the ticket to verify ownership incorrectly relied on a `O(N)` query pattern (`getSupportTickets({ limit: 10000 })`), which was highly inefficient and non-scalable compared to direct lookups.
**Prevention:** Always parse and explicitly `.omit()` restricted fields from incoming `req.body` using a Zod schema (`insertSupportTicketSchema`) on creation and updates. Implement authorization logic by checking ownership via an efficient `getSupportTicket(id)` method before applying modifications, and strictly block non-admin users from altering `userId` and `status`.
## 2026-05-02 - Fixed Authorization Bypass in Support Tickets
**Vulnerability:** Insecure Direct Object Reference (IDOR) / Mass Assignment where any authenticated user could use the `PATCH /api/support/tickets/:id` endpoint to modify any support ticket, and escalate priority or change ownership.
**Learning:** The previous implementation lacked an ownership check (`ticket.userId === req.user.id`) for non-admin users. It was evaluating `isAdmin` but only for subsequent use, skipping the check to restrict modifications on a ticket-by-ticket basis. Additionally, missing input validation on updates allowed unprivileged users to maliciously change ticket assignment properties.
**Prevention:** Always verify object ownership for standard users before performing updates. Use Zod schemas with `.omit()` or `.pick()` on the payload strictly filtering out fields like `userId`, `priority`, or `status` that non-admins should not be able to change.
## 2024-05-06 - [Admin Input Validation & Mass Assignment]
**Vulnerability:** Admin user input on the `PATCH /api/support/tickets/:id` endpoint was being directly passed to the database update query (`updates = req.body`) without passing through the Zod schema validation. This allowed for mass assignment and unvalidated arbitrary input.
**Learning:** Even when users are authorized to perform high-privileged actions (e.g. admins), their input must still be strictly validated against the expected schema to prevent application-level and data-integrity vulnerabilities.
**Prevention:** Always validate all user input regardless of their authorization level. Even admin inputs should be processed with `.parse(req.body)` using the corresponding Zod schema to filter out unexpected fields and perform type assertions.
